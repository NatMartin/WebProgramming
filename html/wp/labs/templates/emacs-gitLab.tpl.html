Emacs

Loading emacs is simple. Ubuntu is a Debian based version of Linux, so it has an aptitude loader. To load Emacs, use the command sudo apt-get install emacs24. The work sudo indicates that you are doing the command as a super user. This needs to be done to load new software, but should be avoided if possible. Working as superuser can damage the software. The command you are executing as superuser is apt-get which updates the software. The apt-get command you are running is install, which indicates that you want to install a new piece of software. Finally, you indicate that the software you are installing is Emacs (emacs24 refers to the version of Emacs, 24 as of this writing).

  1. "sudo apt-get install emacs24"
  2. Enter your password for your linux computer
  3. Answer 'y' when it asks if you want to install it.
  4. "emacs"
  5. Read the emacs tutorial to learn how to move around in the editor

Thats all there is to it. Now when you type emacs into the terminal window, an emacs window will open up.

Git

Git is a version control system. It will keep track of the changes you make to your program and let you revert back to previous versions. This is very important. Code only breaks when you change it, so, if it used to work, you can fix it by changing to back to the way it was when it worked. An empty program works--it does nothing, but it does it very well. So, before you wrote anything your code worked. As you go along, parts of the code will work. When you change it, it may stop working. You will be very happy to be able to return to the state when it worked rather than floundering around trying to figure out which of the things you did made it stop working.

Download Git

As with emacs, the first thing you need to do is download the git program. This is easy to do.

  1. sudo apt-get install git
  2. Enter your password for your linux computer
  3. Answer 'y' when it asks if you want to install it.
  4. git --version
  5. nat@Linus:~$ git --version git version 1.9.1
  6. Read Getting Started

Initialize Git

Git is a little more complicated to use than Emacs. You need to tell it who you are and how to get in touch with you because it can keep track of multiple people working on the same program. If it does not know who you are, it does not know who make the changes. If it does not know who made the changes, you cannot tell who needs to fix things if those changes made things go bad.

You need to tell Git your name an email addres. You can also tell it that you prefer to use Emacs. If you do not, it will use Vim.

  1. git config --global user.name "<your name>"
  2. git config --global user.email <your-email>
  3. git config --global core.editor emacs

Setting up a repository

Once you have let Git know who you are, you can set up a repository. We will create a new directory in your home directory called wp for Web Programming (i.e. ~/wp) and set it up as a Git repository. We will use that directory for the work we do in this class.

  1. cd
  2. mkdir md
  3. cd md
  4. git init
  5. nat@Linus:~/dev$ git init Initialized empty Git repository in /home/nat/md/.git/

Putting a file in Git

So far, we don't have anything in our directory, so there is nothing to store in Git. Let's make a file and put it in the Git repository.

  1. cd ~/md
  2. emacs README
  3. Using emacs, write the following lines, then save the file using c-x c-s.
    This is the directory for Web Programming. It contains the my individual files for the class.
  4. git add README
  5. git commit -m 'initial project version'
  6. nat@Linus:~/md$ git commit -m 'initial project version' [master (root-commit) 36601df] initial project version 1 file changed, 2 insertions(+) create mode 100644 README

Whenever you add a new file to the project, you will need to add it to the list of files that are being tracked using git add. Whenever you change the file, you will need to put it in the repository using git commit. The -m flag we used here adds an initial message. In most cases, it will open an editor so you can explain what changes you made. It may seem silly, but believe me you will not remember what you did in this particular version tomorrow.

Checking Status

You will want to check what you have backed up to Git and the status of the files in the directories protected by Git. The status command does this.

  1. git status
  2. nat@Linus:~/md$ git status On branch master nothing to commit, working directory clean

You are on the master branch, and there is nothing to commit because you have not changed any files in the directory. Let's change the README file and see what happens.

  1. Edit the README file to add the lines:
    Contents: README
    Edit a new file with the name "test".
  2. git status
  3. nat@Linus:~/md$ git status On branch master Changes not staged for commit: (use "git add <file>..." to update what will be committed) (use "git checkout -- $ltfile>..." to discard changes in working directory) modified: README Untracked files: (use "git add <file%gt..." to include in what will be committed) test no changes added to commit (use "git add" and/or "git commit -a")

You can see that now it says that README has been modified. and that there is a new file that has not been included in the project.

Next we will stage the README file to be committed, using git add README (The add command adds the file to the project and indicates that you want to save this version of the file when you do the next commit). Finally, we will remove the test file.

  1. git add README
  2. git commit
  3. Emacs will open with the following information.
    # Please enter the commit message for your changes. Lines starting # with '#' will be ignored, and an empty message aborts the commit. # On branch master # Changes to be committed: # modified: README # # Untracked files: # test #
  4. Add the message "Testing commit." Save the file (c-x c-s) and quit emacs (c-x c-c).
  5. nat@Linus:~/md$ git commit [master 6f66643] Testing commit 1 file changed, 3 insertions(+), 1 deletion(-)
  6. rm test
  7. git status
  8. nat@Linus:~/md$ git status On branch master Untracked files: (use "git add <file>..." to include in what will be committed) test nothing added to commit but untracked files present (use "git add" to track)
