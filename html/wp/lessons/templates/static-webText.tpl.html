Original Intent

Initially, the web supported scientists sharing papers. It was designed and developed by Tim Berners-Lee as a hypertext system for the Internet. Indeed, the first specification for HTTP only included the GET command.

Hypertext and the Internet

Hypertext first appeared in an article by Vannevar Bush in the Atlantic magazine in 1945. Bush described Memex, a collective memory that was stored on microfilm, but could be accessed through a screen on the desk. Each article was linked to other articles in the desk by references. Memex could incorporate new information in the form of additional microfilm rolls, ordered delivered by mail. It, Bush argued, provided access to information in a way that was more attuned to the way people think than traditional libraries because the references were connected to arbitrary microfilm frames and could connect to other arbitrary microfilm frames. Memex included no modern computer technology because the term "computer" still referred to people who performed calculations. Turings paper describing machines that performed symbolic though was still a year in the future.

Early computers cost to much to support the individual use required by hypertext. An early hypertext system was presented in Douglas Englebart', "The Mother of all Demos" in 1968. In the early 1970's Xerox PARC and Carnegie Mellon released hypertext systems. Apple began selling HyperCard, the first widely successful hypertext system, in 1987. Hypercard only ran on a single computer, thought, it did not have access to any information that was not on the computer itself. Ironically, the user interface of the Apple Macintosh, which ran Hypercard, was based on the Lisp Machines of the early 1980's, all of which supported TCP/IP. However, the Macintosh followed in the steps of the IBM PC and the Apple II, which grew out of inexpensive hobbyist computers and did not have networking capability. Steve Jobs decided to leave out networking.

TCP/IP had been in use since the 1970's and in 1983, it became the only protocol on ARPANet. ARPANet was superceeded by NSFNet, from the U.S. National Science Foundation, which, in turn, was replaced by the commercially funded Internet. By the late 1980's all universities and many companies were connected to the Internet and email was the preferred means of communicating in the academy.

Tim Berners-Lee's insight was to combine the Internet with Hypertext. His solution was to provide an HTML tag, called an anchor, that allowed people to insert links to other Web documents. When one clicks on an anchor, the page to which it refers loads in the browser providing a connection between any document on the Web and any other document on the Web.

Sharing Documents

The Web was designed by academics to share documents. In as sense, it grows naturally out of academic writing where every factual statement must refer to the document that demonstrates that the statement is indeed fact or must be proved in the paper itself. The Web only made the standard academic process of tracking down citations much quicker using computers: one need only click on the link and the cited paper would be delivered to your computer screen.

HTML for Document Sharing

HTML include tags and attributes. HTML tags are items that appear in angle brackets such as <p>, which indicates the start of a paragraph. Most tags are paired, including and start tag (e.g., <p>) and an end tag (e.g., </p>).

HTML attributes appear within the brackets of the start tag. For example, the anchor tag, <a>, has an attribute, href that indicates what indicates which pages to load when the reader clicks on the text. For example, if you include <a href="google.com">Google<a> in your document, the text "Google" will appear as a link and you will be taken to the Google home page if you click on it.

An HTML document is specified by the tag <html>. It comes in two parts: the head, specified by the tag <head>, and the body, specified by the tab <body>. The body contains almost all of the text that the browser displays.

The body contains two types of elements: block and inline elements. Block elements are areas on the page whereas inline elements are sequences of text. For example, a paragraph is a block element whereas an anchor is an inline element. Inline elements must be inside block elements.

The HTML below describes a complete web page. It first indicates the type of HTML it is using (HTML5 in this case). The head has only a title in it which will appear on the tab in the browser. The body includes a level 1 header, "Example," then a paragraph which includes a strongly emphasized word.

<!DOCTYPE html> <html> <head> <title>Example</title> </head> <body> <h1>Example</h1> <p> Hello, <strong>World!</strong> </p> </body> </html>
It is a small web page, but shows the general outline.

HTML ignores the indentation and white space in the file, but the indentation is important. It helps the author keep track of the various parts of the document. Each block element should be on a new line, indented to make clear where the block begins and ends. Emacs will help you do this. When you hit the tab-key on a new line, Emacs will indent the line in accordance with the HTML that has come before. This indentation quickly shows you when you have left out a tag. When then indentation does not match what you expected, go back to figure out why your understanding of the file and Emacs' differ.

DOM description of the Document

The Document Object Model (DOM) is the three of HTML element that is specified by the HTML document. It is what is shown on the browser. It allows one to refer to particular parts of the document when using scripting languages. You can specify a name for any element in the DOM by adding the attribute id to any HTML tag. You can specify a name for a group of elements in the DOM by using the attrigute class.

For example, the code below has ids and classes added to the tags.

<!DOCTYPE html> <html> <head id="docHead" > <title id="docTitle" class="headItem" > Example </title> </head> <body id="docBody" > <h1 id="docHeader" class="bodyItem" > Example </h1> <p id="docPg" class="bodyItem" > Hello, <strong>World!</strong> </p> </body> </html>
The head is name "docHead", the body is name "docBody". All of the tags in the head are added to the class "headItem" whereas all of the items in the body are added to the class "bodyItem". Objects may belong to more than one class (the classes are listed in the quotation marks separated by spaces. (e.g. class="headItem bodyItem")

CSS for Document Sharing

CSS (Cascading Style Sheets) is a language for specifying the appearance of HTML pages. It allows the separation of the content of the document from appearance of the document, allowing the author to concentrate on one thing at a time.

CSS allows one to specify the way a particular tag will look. For example, you can change the way to different headers look or you can specify that strong emphasis should be red text. CSS directives are implemented in the browser like HTML, so the browser still had leeway to implement them.

Writing Documents for the Web

HTML should only indicate what the elements in the document are. That is, it should explain what the document elements are, not how they should appear on the screen.

For example, there are two tags <strong> and <b>. The first indicates that the author wants strong emphasis of the enclosed text, the second says the author wants the enclosed text to be bold. Usually, both commands change the text's appearance in the same way. However, <strong> should always be preferred.

There are three reasons for this:

  • The browser knows what kind of device it is on: the size and resolution of the screen, the colors it can display, etc. It can make better decisions about how best to display the document if it knows the authors intention. If it only knows how the author wants it to be displayed it cannot make intelligent choices.
  • If you format your documents using tags that tell what the elements in the document are, you can change the appearance of these items using CSS. For example, if you indicate strong emphasis, you can make red the text with strong emphasis. It makes less sense to change bold to red, and if you actually need to have both bold and strong emphasis, you can change one without changing the other.
  • You know what kind of document element you are writing, but you cannot what it should look like until you see it. On the Web, you only see the appearance of a document after you put it on the web.

<div> and <span>

Two of the most useful commands in HTML, <div> and <span>, do nothing in themselves. Instead, they allow the user to specify certain elements of the code so these elements can be modified using CSS. In essence, these commands allow you to generate your own elements. <div> is a block tag and <span> and inline tag.

When writing HTML, focus on the meaning of the text not the appearance. Put the elements in appropriate tags. HTML will give you the basic tools to break up your work. You can put in headings and illustrations and captions, but some elements are specific to your writing. These is where <div> and <span>. You can use these tags to call out particular elements in your writing. For example, you may want to specify a particular topic. To do this, enclose those elements with <div> tags and give it a class name. The class name lets you call out a particular set of elements in your text. Later you can changing the styling of these elements without changing any of the other elements. If you want to specify a particular sequence of works use the <span> tags.

One writes web pages to communicate. Use the <div> and <span> tags to indicate what the section is trying to communicate (E.g., a reference, a piece of code, a set of instructions). You can later go back and add formatting to make the intention clearer in the Web browser

The Limits of Writing Documents

Initially, Web documents were written by authors to share information, but the effort required to maintain a set of documents quickly became apparent. People developed programming tools to help manage web sites.

In addition, tools for accepting input a providing that input to programs were developed. Eventually, the Web took over most client-server model applications. The browser is a container on the client computer that can be programmed using HTML, CSS, and JavaScript to write the client side more easily. Web server programs provide containers on the server side to do the same thing using a multitude of languages.
